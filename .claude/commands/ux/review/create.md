# Create a UX review of the desktop app

Run a full UX review of the Photosphere desktop app as if you were a brand-new
user opening it for the first time. Capture screenshots, look for problems, and
write up the findings.

## Output rules

- Put ALL output in the `ux-review/` directory at the repo root.
- `ux-review/` is gitignored. Do NOT commit anything in it.
- Write the review to `ux-review/review.md`.
- Screenshots go in `ux-review/screenshots/` (the capture script does this).

## Writing style for the review

- Plain English. No jargon. No made-up terms.
- No waffle. Only say what matters.
- Use bullet points wherever you can.
- Keep each finding short and easy to understand.

## Steps

### 1. Capture the screenshots

Run this from the repo root:

```bash
bun run screenshots
```

- It bundles the app, launches it headless (via `xvfb-run` on Linux), opens the
  50-asset test database, and walks through the main screens.
- It writes PNGs to `ux-review/screenshots/`.
- The script is `apps/desktop/screenshots/capture-ux.sh`. It drives the app
  through the test control server (see `docs/testing/screenshots.md`).
- If you need extra screens, add steps to that script, then re-run.

How the capture works (for reference):

- The app starts in test mode (`PHOTOSPHERE_TEST_MODE=1`), which starts a small
  HTTP control server.
- The script sends it commands: open a database, go to a page, click a thing,
  take a screenshot.
- A screenshot is taken by `POST /screenshot`, which calls Electron's
  `webContents.capturePage()` and saves a PNG.
- Gotcha: gallery thumbnails need `/long-press-click`, not `/click` (a plain
  click is ignored).

### 2. Look at every screenshot

- Read each PNG in `ux-review/screenshots/`.
- Judge each screen as a first-time user would:
  - Is it clear what this screen is for?
  - Is it clear what to do next?
  - Is anything confusing, hidden, or missing?
  - Are buttons and icons understandable without labels?
  - Are empty states helpful or just blank?
- When something looks wrong, check the code to confirm it is real (not just a
  capture glitch). Frontend code is in `packages/user-interface/src` and
  `apps/desktop-frontend/src`.

### 3. Known things to check (from past reviews)

Re-check these each time, but judge them fresh. They may be fixed by now:

- First-run screen: does it explain what the app is and what to do? Is the word
  "database" too technical for a photo app?
- Opening a photo, then clicking "Gallery": does it return to the grid, or stay
  stuck on the photo? (The viewer is driven by state, not the route.)
- The three-dot button: does it open a search/sort panel? That icon usually
  means an overflow menu, so it can be confusing.
- The right-hand panel: can it be closed the same way it was opened? Does it
  cover other pages after you navigate away?
- Empty tables (Databases, Secrets) and the Map: do they show helpful text when
  empty, or just blank?
- Photo toolbar icons: do they have labels or tooltips?

Do not assume these are still problems. Look and confirm.

### 4. Write the review

Write `ux-review/review.md` with:

- A short summary (a few lines).
- Findings grouped by screen or area.
- A severity tag on each finding:
  - High: blocks or confuses a new user, or is a clear defect.
  - Medium: friction worth fixing.
  - Low: small polish.
- For each finding, link the screenshot that shows it, e.g.
  `![Gallery](screenshots/02-gallery.png)`.
- A short, ordered list of the top things to fix.
- A note that screenshots live in `ux-review/screenshots/` and that the review
  is regenerated by `/ux:review:create` and is not committed.

### 5. Finish

- Tell the user the review is written to `ux-review/review.md`.
- Give a 3-5 bullet summary of the biggest findings.
- Remind them they can walk through it with `/ux:review:walkthrough`.
- Do not commit anything.
